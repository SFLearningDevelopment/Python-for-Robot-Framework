In Module 1 you saw one Python function become one keyword. Now you'll learn the Python that goes inside those functions — variables, arguments, conditionals, and loops — with every example written the way a tester writes it: a function that returns a value your .robot test then checks. Keep the same two files from Module 1 handy (a .py library and a .robot test); you'll add to them as you go.

Variables, types, and the return value

A variable is a name for a value. Python figures out the type from what you assign: text is a str, whole numbers are int, decimals are float, and true/false is a bool. For a tester, the value that matters most is the one a function hands back with return — that is what crosses the bridge into your test and becomes a Robot ${variable}.

CheckoutLibrary.py
# Returns a float — the test will capture it as ${total}.
def order_total(price, quantity):
    # Robot passes both as strings; convert to the right types first.
    return float(price) * int(quantity)

Refresh your knowledge? — what a variable is

A variable is a labelled box that holds a value, created by assigning with =, as in price = 19.99. The name on the left lets you reuse the value on the right without repeating it. Python is dynamically typed: you don't declare the type, it's inferred from the value, and the same name can later hold a different type. Common built-in types are str (text, in quotes), int (whole numbers), float (decimals), and bool (True/False).

Functions, arguments, and defaults

A keyword's arguments are just the function's parameters. Python lets you give a parameter a default, so callers can omit it — and in Robot Framework that becomes an optional keyword argument. You can also pass arguments by name from a test using name=value syntax, which reads clearly for anyone maintaining the suite.

CheckoutLibrary.py (add this)
def apply_discount(price, percent=0):
    # percent defaults to 0 — no discount unless the caller passes one.
    price = float(price)
    percent = float(percent)
    return round(price - (price * percent / 100), 2)
checkout_tests.robot (add this)

    ${final}=    Apply Discount    100
    Should Be Equal As Numbers    ${final}    100.0

    ${final}=    Apply Discount    100    percent=10
    Should Be Equal As Numbers    ${final}    90.0

The first test omits the discount and gets the default; the second names the argument percent=10. Same keyword, two call styles — this is how one Python function serves several test cases.

Refresh your knowledge? — function arguments

Arguments are the inputs a function receives. In def apply_discount(price, percent=0):, price is required and percent has a default of 0, so it's optional. You can pass arguments positionally (in order) or by keyword (percent=10, naming them). Defaults make a function flexible: callers supply only what differs from the common case. Robot Framework maps all of this directly onto keyword calls.

Conditionals: returning a verdict

Tests constantly ask "does this value meet the rule?" In Python you express that with if/elif/else, returning a result the test can act on. Comparisons (==, !=, >, <, >=, <=) produce a bool, and a function that returns a bool pairs naturally with Robot's Should Be True.

CheckoutLibrary.py (add this)
def shipping_band(weight_kg):
    weight = float(weight_kg)
    if weight <= 1:
        return "light"
    elif weight <= 10:
        return "standard"
    else:
        return "heavy"
checkout_tests.robot (add this)

    Should Be Equal    ${band}    light

    Should Be Equal    ${band}    standard

The keyword returns one of three strings depending on the input; each test asserts the band it expects. The decision logic lives in Python, and the test just states the expected outcome.

Refresh your knowledge? — booleans and comparisons

A comparison asks a yes/no question and evaluates to a boolean: 5 > 3 is True, 2 == 4 is False. The operators are == (equal), != (not equal), and >, <, >=, <=. An if statement runs its block only when its condition is True; elif checks a further condition if earlier ones were False; else catches everything remaining. Note the double == for comparison versus single = for assignment.

Loops: building a result from many items

When a test works over many values — several prices, a batch of records — a loop lets one keyword process them all and return a single summarizing result. for repeats once per item in a collection; while repeats as long as a condition holds. Most test logic uses for.

CheckoutLibrary.py (add this)
def cart_total(prices):
    # prices arrives as a Python list from the test.
    total = 0.0
    for price in prices:
        total = total + float(price)
    return round(total, 2)
checkout_tests.robot (add this)
*** Test Cases ***
Cart Total Sums All Items
    ${prices}=    Create List    10.00    5.50    4.50
    ${sum}=    Cart Total    ${prices}
    Should Be Equal As Numbers    ${sum}    20.0

The built-in keyword Create List builds a Robot list; the test passes it to Cart Total, whose for loop adds every item and returns the sum. One keyword, any number of prices.

Refresh your knowledge? — iteration

Iteration means doing something once for each item in a collection. for price in prices: takes each element of prices in turn, binds it to price, and runs the indented block. A running variable (here total) accumulates a result across the passes. A while loop instead repeats as long as a condition stays True, so you must ensure the condition eventually becomes False or the loop never ends.
