A library as a Python class

A class groups related keywords and can hold state between calls within a suite. Each method becomes a keyword. Setting ROBOT_LIBRARY_SCOPE = 'SUITE' means one instance is shared across the whole suite, so state (like a running tally) persists from one test to the next.

CartLibrary.py
class CartLibrary:
    ROBOT_LIBRARY_SCOPE = 'SUITE'   # one shared instance per suite

    def __init__(self):
        self._items = []            # state kept across keyword calls

    def add_item(self, name, price):
        self._items.append((name, float(price)))

    def item_count(self):
        return len(self._items)

    def cart_total(self):
        return round(sum(price for _, price in self._items), 2)

Both Add Item calls write into the same shared instance, so Item Count and Cart Total see both items. A class turns a set of related keywords into one cohesive library.

Refresh your knowledge? — classes and objects

A class is a blueprint that bundles data and the functions that work on it. Creating one gives you an object (an instance). Methods are functions defined inside the class; the first parameter, self, refers to the instance, so self._items is data that belongs to that object. __init__ runs once when the object is created, setting up its starting state. In a Robot library, each method becomes a keyword and self lets those keywords share state.

The robot.api toolkit: names, tags, and logging

Robot Framework gives your Python two useful tools. The @keyword decorator sets a custom keyword name and tags, so the name in tests reads well regardless of the method name. The logger writes messages into the test log at different levels, which is how a keyword explains what it did.

CartLibrary.py (add these imports and method)
from robot.api.deco import keyword
from robot.api import logger

# ...inside the CartLibrary class...

    @keyword('Apply Coupon Code', tags=['pricing'])
    def apply_coupon(self, code):
        discount = {'SAVE10': 0.10, 'SAVE20': 0.20}.get(code, 0)
        logger.info(f"Coupon {code} applied {int(discount*100)}% off")
        total = self.cart_total()
        return round(total * (1 - discount), 2)
cart_tests.robot (add this)
Coupon Reduces The Total
    Add Item    Book    100.00
    ${final}=    Apply Coupon Code    SAVE20
    Should Be Equal As Numbers    ${final}    80.0

The method apply_coupon is called in the test as Apply Coupon Code — the name set by the decorator, not the method name. Its logger.info(...) line appears in log.html, so anyone reading the report can see which coupon was applied.

Refresh your knowledge? — importing modules

An import statement brings code from another module into yours. from robot.api.deco import keyword pulls in just the keyword decorator; from robot.api import logger pulls in the logger object. A decorator, written with @ above a function, wraps it to add behaviour — here, @keyword('Apply Coupon Code') tells Robot Framework to expose the method under that exact name instead of the auto-generated one.

A structured suite: setup, teardown, resources, and tags

A maintainable suite separates reusable steps into a resource file, runs shared preparation with Suite Setup and cleanup with Suite Teardown, and labels tests with tags so you can run a subset. Variables declared in *** Variables *** keep configuration in one place.

cart_keywords.resource
*** Settings ***
Library           CartLibrary.py

*** Keywords ***
Start With An Empty Cart
    Log    Starting a fresh cart for this suite

Add Two Sample Items
    Add Item    Book    12.00
    Add Item    Pen    3.50
suite_tests.robot
*** Settings ***
Resource          cart_keywords.resource
Suite Setup       Start With An Empty Cart

*** Variables ***
${EXPECTED_COUNT}     ${2}

*** Test Cases ***
Sample Cart Has Expected Count
    [Tags]    pricing    smoke
    Add Two Sample Items
    ${count}=    Item Count
    Should Be Equal As Integers    ${count}    ${EXPECTED_COUNT}

Run only the smoke tests with robot --include smoke suite_tests.robot. The Resource import pulls in the shared keywords; Suite Setup runs once before the tests. As a rule, prefer explicit wait keywords over fixed Sleep calls — a wait proceeds as soon as the condition is met, while a sleep always costs the full delay.

Refresh your knowledge? — test suite organization

A suite is a .robot file (or folder of them) holding related tests. Suite Setup and Suite Teardown run once before and after all tests in the file; Test Setup/Test Teardown run around each test. A resource file holds shared keywords and imports you reuse across suites via the Resource setting. Tags label tests so you can include or exclude groups at run time with --include / --exclude.

Putting it together: a browser test

Your custom library and an ecosystem library can work in the same suite. SeleniumLibrary adds browser keywords like Open Browser, Input Text, and Page Should Contain. Install it first, then run a small end-to-end test. This is the one example that reaches outside Python into a real browser.

  1. Install the library and a browser driver: pip install robotframework-seleniumlibrary. Selenium 4 manages drivers automatically for recent Chrome or Firefox; make sure the matching browser is installed.
  2. Create browser_tests.robot in your module folder with the code below.
  3. Run it with robot browser_tests.robot. A browser window opens, the test drives it, and the result appears in report.html.
browser_tests.robot
*** Settings ***
Library           SeleniumLibrary
Library           CartLibrary.py
Suite Teardown    Close All Browsers

*** Variables ***
${URL}        https://example.com
${BROWSER}    chrome

*** Test Cases ***

    Open Browser    ${URL}    ${BROWSER}
    Page Should Contain    Example Domain
    # ...and your own keyword in the same test.
    Add Item    Book    12.00
    ${count}=    Item Count
    Should Be Equal As Integers    ${count}    1
    [Teardown]    Close Browser

One test uses Open Browser and Page Should Contain from SeleniumLibrary and Add Item and Item Count from your own CartLibrary — the ecosystem library and your Python library side by side. That is the whole course in one file: Python you wrote, driven as keywords, inside a real Robot suite. You're ready for the certification exam.

Refresh your knowledge? — reading the robot output

Every run writes three files. report.html is the high-level pass/fail summary with timing and tags. log.html is the step-by-step detail — expand a keyword to see its arguments, return value, and any logger messages, which is where you diagnose a failure. output.xml is the machine-readable source the other two are built from, useful for CI dashboards. When a test fails, open log.html and find the first red keyword; its message usually names the cause.
