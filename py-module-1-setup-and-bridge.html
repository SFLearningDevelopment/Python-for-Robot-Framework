Setup and the Python↔Robot bridge

robotframework 7.4.2 python 3.8+ first keyword
Versions accurate as of 1 July 2026. Robot Framework 7.4.2 requires Python 3.8 or newer. Run robot --version after installing and check PyPI if you need the current release.

Your goal in this module is small and concrete: get a working environment, run a real Robot Framework test, and prove the one idea the whole course rests on — that a plain Python function can become a Robot keyword you call from a .robot file. Once you can see your own Python running as a keyword, everything after this is just adding depth.

You will not write much Python here. You will install the tools, run a test that already works, then write four lines of Python and watch Robot Framework call them. That connection is the through-line of the course.

Where Python fits

Robot Framework is a test automation framework built in Python. You write tests in a readable, table-style syntax made of keywords. Some keywords are built in (Log, Should Be Equal); others come from libraries. When the built-in and library keywords aren't enough, you write your own — and the primary language for writing them is Python. That is where you come in as a tester: Python is how you extend Robot Framework with logic specific to your system under test.

Refresh your knowledge? — what a test automation framework is

A test automation framework gives you a structure for writing, organizing, and running tests automatically instead of clicking through an application by hand. It handles the repetitive parts — running each test, recording pass or fail, and producing a report — so you can focus on describing what to check. Robot Framework's particular style is keyword-driven: each test is a sequence of named actions (keywords) that read almost like plain instructions.

You read keywords top to bottom and can tell what a test does without knowing any code. Some are built into Robot Framework (like Log and Should Be Equal); others come from a library you import (like Open Browser). Here are a few common ones, each shown in use:

Log
Writes a message to the test log. Log    Order total was 250
Should Be Equal
Fails the test unless two values match. Should Be Equal    ${status}    PASS
Open Browser
Opens a browser at a URL (from a library). Open Browser    https://example.com    chrome
Input Text
Types text into a field found by a locator. Input Text    id=username    demo
Click Button
Clicks a button found by a locator. Click Button    id=login
Page Should Contain
Fails unless the given text appears on the page. Page Should Contain    Welcome back

Install and verify

Robot Framework installs from PyPI with pip. Installing it also gives you the robot command that runs your tests. Confirm the version before you go further — a mismatched Python or Robot version is the most common reason a first test won't run.

terminal
# Check your Python — must be 3.8 or newer
python --version

# Install Robot Framework (also installs the `robot` command)
pip install robotframework

# Confirm the framework installed
robot --version
# e.g. Robot Framework 7.4.2 (Python 3.12.x on ...)

Later in the course you'll use an ecosystem library for browser testing. You don't need it yet, but this is how such a library installs — as a separate pip package that adds more keywords:

terminal
# You will use this in Module 4 — no need to run it now
pip install robotframework-seleniumlibrary
Refresh your knowledge? — pip and virtual environments

pip is Python's package installer; it fetches libraries from PyPI (the Python Package Index) and makes them available to your Python. A virtual environment is an isolated Python setup for one project, so its packages don't collide with another project's. Create and activate one with python -m venv venv then source venv/bin/activate (macOS/Linux) or venv\Scripts\activate (Windows). Installing Robot Framework inside an activated venv keeps this course's tools separate from everything else on your machine.

Your first .robot file

A Robot Framework test lives in a plain-text .robot file organized into sections marked by *** ... *** headers. The three you'll use constantly are Settings (imports and configuration), Test Cases (the tests themselves), and Keywords (your own reusable actions). Here is a complete, runnable test that uses only built-in keywords.

first_tests.robot
*** Settings ***
Documentation     My first Robot Framework suite.

*** Test Cases ***
Greeting Is Logged
    Log    Hello from Robot Framework
    Should Be Equal    ${1}    ${1}

Notice the spacing: Robot Framework separates a keyword from its arguments with two or more spaces. A single space is read as part of the same value. This trips up almost everyone once. Run the file from the terminal:

terminal
robot first_tests.robot

Robot Framework prints a pass/fail summary and writes three files into the current folder: log.html (step-by-step detail), report.html (high-level result), and output.xml (machine-readable). Open log.html to see your Log message and each keyword's status. Reading these reports is how you'll diagnose every future test.

Refresh your knowledge? — the command line

The command line (terminal on macOS/Linux, Command Prompt or PowerShell on Windows) lets you type commands instead of clicking. You run a program by typing its name and pressing Enter; cd foldername moves into a folder, and ls (macOS/Linux) or dir (Windows) lists what's there. Robot Framework tests are launched this way with the robot command, so you'll want to cd into the folder holding your .robot file first.

The bridge: a Python function becomes a keyword

This is the core of the course. A Python file is a library; each function in it becomes a keyword whose name is the function name with underscores read as spaces. You import the library in Settings, then call the function as a keyword in a test. Create two files in the same folder.

MathLibrary.py
# Each function here becomes a Robot Framework keyword.
def add_numbers(a, b):
    # Robot passes arguments as strings — convert before adding.
    return int(a) + int(b)
bridge_tests.robot
*** Settings ***
Library           MathLibrary.py

*** Test Cases ***
Adding Two Numbers Works
    ${result}=    Add Numbers    2    3
    Should Be Equal As Integers    ${result}    5

Run it with robot bridge_tests.robot. The function add_numbers became the keyword Add Numbers; the test called it, captured the return value into the variable ${result}, and asserted on it. You just wrote Python and drove it from a Robot test. Every remaining module builds on exactly this move — richer Python, returning richer results, driven by more structured suites.

Refresh your knowledge? — what a function is

A function is a named, reusable block of code that can take inputs (arguments) and hand back a result with return. In def add_numbers(a, b):, the name is add_numbers and a and b are its arguments. Calling add_numbers(2, 3) runs the block and gives back 5. In Robot Framework, that returned value is what a test captures into a ${variable} and checks — so the return statement is the hand-off point between your Python and your test.
