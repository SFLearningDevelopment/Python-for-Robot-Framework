Data structures and test data

lists dictionaries strings & files exceptions
Examples run on Python 3.8+ with Robot Framework 7.4.2. Verify current versions on PyPI before installing.

Real tests move data around: a list of expected rows, a dictionary that looks like an API response, a data file that drives many cases, and an error that should fail a test cleanly. This module covers the four Python data structures a tester reaches for most, each shown feeding a Robot test. Keep a .py library and a .robot test side by side in one folder, as before.

Lists

A list is an ordered collection you can index and slice. Python indexes from 0, so items[0] is the first element. A keyword that returns a list hands Robot a list it can index with ${result}[0].

DataLibrary.py
def top_three(scores):
    # Sort highest-first and return the first three.
    numbers = [int(s) for s in scores]
    numbers.sort(reverse=True)
    return numbers[:3]
data_tests.robot

The slice numbers[:3] takes the first three items; the test reads the highest with ${top}[0] and checks the length with the built-in Length Should Be.

Refresh your knowledge? — what a collection is

A collection holds multiple values in one object. A list is written with square brackets, [10, 20, 30], keeps its items in order, and allows duplicates. You reach an item by position with list[index] (starting at 0), and take a range with a slice like list[:3] (first three) or list[1:] (all but the first). Lists can grow with .append(x) and be ordered with .sort().

Dictionaries

A dictionary maps keys to values — ideal for modelling a JSON-like API response. You read a value by its key, and .get() lets you supply a fallback when a key might be missing. In Robot, a returned dict is accessed with ${response}[key].

DataLibrary.py (add this)
def parse_user(record):
    # record is a dict; .get() avoids a crash on a missing key.
    return {
        "name": record.get("name", "unknown"),
        "active": record.get("active", False),
    }
data_tests.robot (add this)
Parse User Fills Missing Fields
    ${record}=    Create Dictionary    name=Asha
    ${user}=    Parse User    ${record}
    Should Be Equal    ${user}[name]    Asha
    Should Be Equal    ${user}[active]    ${False}

The record has no active key, so .get("active", False) supplies the default. The test reads fields with ${user}[name] — the same bracket access you use for lists, but keyed by name.

Refresh your knowledge? — key–value pairs

A dictionary stores key–value pairs, written with braces: {"name": "Asha", "active": True}. Each key is unique and maps to one value. You read a value with d["name"], but that raises an error if the key is absent; d.get("name", default) returns the default instead of failing. Dictionaries model structured records well, which is why API responses and config are so often dictionaries.

Strings and files

Test data often arrives as text — a name to normalize, or a small data file that drives many cases. Python strings have handy methods (.strip(), .upper(), .split()), and f-strings build readable messages. Reading a file lets one keyword feed a data-driven suite.

users.csv
Asha,active
Ben,inactive
Chen,active
DataLibrary.py (add this)
def active_names(csv_path):
    # Return the names whose status column is "active".
    names = []
    with open(csv_path) as f:
        for line in f:
            name, status = line.strip().split(",")
            if status == "active":
                names.append(name)
    return names
data_tests.robot (add this)
Active Names Are Extracted
    ${names}=    Active Names    users.csv
    Should Contain    ${names}    Asha
    Should Not Contain    ${names}    Ben

line.strip() removes the trailing newline; .split(",") breaks each row into fields. The keyword returns only the active names, and the test asserts membership with Should Contain. Put users.csv in the same folder as the test.

Refresh your knowledge? — strings as sequences

A string is a sequence of characters in quotes. Useful methods include .strip() (remove surrounding whitespace, including newlines), .upper()/.lower() (change case), and .split(",") (break into a list on a separator). An f-string like f"Total: {amount}" inserts a variable's value into text. Reading a file with with open(path) as f: gives you its lines to loop over, and the with block closes the file automatically when done.

Exceptions: failing a test on purpose

When input is invalid, a keyword should fail clearly rather than return a wrong answer. In Python you raise an exception; Robot Framework turns an unhandled exception into a failed keyword, with your message in the log. try/except lets you catch and reshape errors when you'd rather handle them.

DataLibrary.py (add this)
def parse_positive_int(value):
    number = int(value)          # raises ValueError on non-numeric text
    if number <= 0:
        raise ValueError(f"Expected a positive number, got {number}")
    return number
data_tests.robot (add this)
Negative Input Fails The Keyword
    Run Keyword And Expect Error    *positive number*    Parse Positive Int    -5

Valid Input Returns The Number
    ${n}=    Parse Positive Int    7
    Should Be Equal As Integers    ${n}    7

The first test uses Run Keyword And Expect Error to assert that a negative value does fail, matching part of the message. The second confirms a valid value returns normally. Raising an exception is how your Python signals a genuine failure to the test.

Refresh your knowledge? — what an error is

An exception is Python's way of signalling that something went wrong, such as ValueError for a bad value. When code hits an unhandled exception, it stops and reports the error. You can trigger one deliberately with raise ValueError("message"), or catch one with try: / except ValueError: to handle it gracefully. In Robot Framework, an unhandled exception inside a keyword becomes a failed test step — which is exactly what you want when the input is invalid.
